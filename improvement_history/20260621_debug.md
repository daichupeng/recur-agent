## Big Picture — Orchestration, Workflow, Architecture

### A1. No top-level conversation loop

The root orchestrator is a `SequentialAgent` that runs all five phases exactly once in order and then terminates. Once the final `recommend_policies` phase completes, the session ends. There is no mechanism for the user to ask follow-up questions, and the agent cannot re-profile the user after a later phase surfaces new coverage types. The stated requirement explicitly asks for a system that "learns" and "asks follow-up questions as more potential coverages are identified during market research" — this is structurally impossible in a single-pass sequential flow.

### A2. LoopAgent composition has no termination condition

Two orchestrators (`profile_user_insurance_context`, `refine_requirements_with_followups`) use `LoopAgent`. ADK's `LoopAgent` terminates only when a sub-agent triggers an escalation or `max_iterations` is reached. The compiler template (`adk_loop_agent.py.j2`) never emits `max_iterations`, and the atomic agents inside the loops have no escalation logic. Depending on the ADK version, these agents either run infinitely or exit immediately with no useful output.

### A3. No inter-agent data contract — agents are blind to each other's outputs

ADK's `SequentialAgent` threads outputs through session state, but none of the child `LlmAgent` instructions state which session state keys to read from or write to. Each agent receives the raw conversation context with no structured handoff from prior steps. The pipeline's intended accumulation — "profile user → research market → identify gaps → refine → recommend" — is semantically incoherent because no agent knows what the prior one produced.

### A4. Workflow pattern (batch pipeline) is architecturally mismatched to the requirement (dynamic conversation)

The decomposed tree models the system as five fixed sequential phases, each running once. The requirement is a conversational agent that dynamically discovers coverage types and adapts its questions accordingly. A fixed pipeline cannot handle the feedback loops implicit in the requirement: market research results should affect what follow-up questions are asked, which should affect which products are ranked. A coordinator-driven loop with adaptive routing is the correct architecture; the current decomposition produces something structurally closer to a data ETL pipeline.

## Critical Nodes — Local Errors and Bugs

### B1. `_normalise_indent()` produces `IndentationError` in all 7 tool files

All seven DETERMINISTIC_CODE and EXTERNAL_API atomic files have `IndentationError` or `unexpected indent` and crash on import. Root cause is in `src/agents/tool_implementor.py:_normalise_indent()`:

The function calls `textwrap.dedent(body)` then prepends `"    "` to every non-empty line. `textwrap.dedent` strips only the **minimum common leading whitespace** across all lines. When the LLM emits `import` statements at column 0 (module-level style) and the rest of the function body at 4 or 8 spaces, the minimum common indent is 0 — nothing is stripped. The flat prepend then produces:
- `import` statements at 4 spaces (0 + 4)
- body statements at 8 or 12 spaces (4/8 + 4)

inside a function body that requires all outer-level statements at exactly 4 spaces, causing `IndentationError`. All seven tool files exhibit this pattern. The project cannot be imported.

Concrete example (`research_user_suggestion.py`):
```python
    import httpx          # 4 spaces — correct inside function body

        base_url = "..."  # 8 spaces — wrong, should be 4
```

### B2. Generated project venv has a dangling Python symlink — project cannot run

`.venv/bin/python` → `/usr/local/bin/python3`, which does not exist on this machine (Python lives at `/Users/chupeng/.pyenv/versions/3.11.9/bin/python3`). The symlink `python3.11` is also broken. The compiled project cannot be executed without manually deleting and recreating the venv. The platform compiler generates the project folder but neither runs `uv sync` / `pip install` nor emits a setup script. This is a portability failure: any machine where Python is not at `/usr/local/bin/python3` receives an unrunnable project.

### B3. LoopAgent semantics: multiple sub_agents in a loop with no stop signal

Both LoopAgent orchestrators wrap 2–3 sub_agents as a list. ADK's `LoopAgent` runs its `sub_agents` as a sequential pass per iteration, then loops. None of the sub-agents emits an escalation action, and no `max_iterations` is set. This means:
- The loop never terminates from within
- Even if `max_iterations` were set, the sub-agents inside the loop are separate `LlmAgent` instances that do not share state between iterations — there is no accumulated "profile" or "conversation history" building up across loop iterations as the design intends

## Minor Bugs — Non-blocking, Affecting Quality and Performance

### C1. Extra blank lines between imports in all orchestrator templates

The Jinja2 `{% for imp in child_imports %}` ... `{% endfor %}` loop in the composite templates (`adk_sequential_agent.py.j2`, `adk_loop_agent.py.j2`, etc.) emits a blank line after each import statement. Every generated orchestrator file has double-spaced import blocks.

### C2. LLM agent `instruction` embeds the output schema as an inert string — no enforcement

`adk_llm_agent_stub.py.j2` writes the JSON Schema into the `instruction` field and asks the model to "Return a JSON object matching this schema." No `output_schema` / `response_schema` field is set on the `LlmAgent`. Gemini has no enforcement constraint, so it can return free-form text or a differently-structured object; nothing validates or rejects a non-conforming response.

### C3. LLM coordinator template has no `instruction` — routing is undirected

`adk_coordinator_agent.py.j2` generates an `LlmAgent` with `sub_agents` but no `instruction`. The coordinator LLM has no system prompt describing when to invoke `retrieve_user_provided_policies_agent` versus `prompt_user_for_policy_details_agent`. Routing decisions are effectively arbitrary.

### C4. All pipeline agents use `claude-haiku-4-5`, a model less reliable for code generation

`BaseAgent.MODEL = "claude-haiku-4-5"` is hardcoded. Every pipeline agent — Decomposer, SchemaArchitect, ToolImplementor, ComplexityReviewer — uses Haiku. Haiku's lower reliability for multi-line structured code generation is a direct contributing factor to the mixed-indentation defect in B1. No per-agent model override mechanism exists.

### C5. Tool atomic `LlmAgent` wrapper has no `instruction` — tool is called blindly

`adk_tool_stub.py.j2` wraps every tool in an `LlmAgent` that exposes the tool via `tools=[...]`. The wrapper `LlmAgent` has `description` but no `instruction`, so the LLM driving it has no guidance on what arguments to pass to the tool, what to do with the result, or how to format its response to downstream agents.
