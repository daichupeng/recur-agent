## P1 — Foundational

### P1.1  Teach the Decomposer ADK-specific composition rules

**Solves:** A1, A3, D4

The Decomposer's system prompt defines composition types abstractly. It must instead encode the ADK runtime constraints that determine when each type is correct and what the generated code will look like.

Add a `## ADK constraints` section to `decomposer.py`'s `_SYSTEM_PROMPT`:

```
## ADK runtime constraints (you MUST follow these)

SEQUENTIAL — use by default for any ordered multi-step flow.
  Generated as: SequentialAgent(sub_agents=[...])

PARALLEL — use only when children are fully independent (no data dependency between them).
  Generated as: ParallelAgent(sub_agents=[...])

LOOP — use ONLY when:
  (a) exactly ONE child agent is being repeated, AND
  (b) that child agent can detect a terminal condition and signal it via ADK escalation.
  Never use LOOP for multi-turn conversation unless the child handles both the turn and
  the exit condition internally. If unsure, use SEQUENTIAL with a dedicated termination step.
  Generated as: LoopAgent(sub_agents=[one_agent], max_iterations=N)

LLM_COORDINATOR — use for routing/dispatch where the decision is ambiguous at design time.
  The coordinator is an LlmAgent that routes to children based on context.
  Its instruction MUST describe each child and when to invoke it.
  Generated as: LlmAgent(sub_agents=[...], instruction="Route based on ...")
```

This is a prompt-only change to `decomposer.py`. It prevents wrong composition types from entering the tree; the template layer then renders correct code for whichever type was chosen.

### P1.2  Extend `SkillNode` with session-state fields and an `instruction` field

**Solves:** D1, D3

`SkillNode` is missing two fields that block the entire LLM-agent generation chain:

- **`state_reads: list[str]`** — ADK session state keys this node reads from.
- **`state_writes: list[str]`** — ADK session state keys this node writes to.
- **`instruction: Optional[str]`** — Engineered system prompt, used by `adk_llm_agent_stub.py.j2` and `adk_tool_stub.py.j2` wrapper agents.

Add to `src/orchestrator/state.py`:

```python
state_reads:  list[str] = Field(default_factory=list)
state_writes: list[str] = Field(default_factory=list)
instruction:  Optional[str] = None
```

Populate `state_reads`/`state_writes` in the Decomposer output schema (alongside `composition_type`) so they are set at decomposition time, not inferred later. Use them in `adk_llm_agent_stub.py.j2` to tell the LLM which session state keys to read and write, replacing the current schema-as-string approach.

Add a `PromptEngineerAgent` (planned in `20260619_solution.md` as P3, now promoted) that generates `instruction` for each LLM atomic node using `name`, `description`, `input_schema`, `output_schema`, `state_reads`, and `state_writes`. Called immediately after schema hydration, before compilation.

### P1.3  Granular HITL: per-node edit and hint-driven re-decompose

**Solves:** A4, D5

The binary approve/rollback model is the primary reason bad decompositions accumulate in the tree. Replace it with three HITL actions:

| Action | What happens |
|--------|-------------|
| **Approve** | Layer accepted; proceed to schema hydration (current behaviour) |
| **Rollback** | Entire layer discarded; Decomposer retries from scratch (current behaviour) |
| **Edit node** | Human edits a single node's description or `composition_type` in the UI; pipeline re-decomposes only that subtree with the new description as context, leaving sibling nodes untouched |

Implementation:
- Add `events.edit: asyncio.Queue` to `PipelineEvents`, carrying `(node_id, new_description, hint)`.
- In `pipeline.py`, add a third branch in the HITL wait: if an edit arrives, call `decomposer.decompose([target_node])` with the edited description and a hint string injected as a user-turn prefix. Only the target node's subtree is rebuilt.
- The UI gains a per-node edit form next to each node in the dashboard.

This makes correction cheap — fixing one wrong `composition_type` no longer costs a full layer retry.

### P1.4  Add a second HITL checkpoint after schema hydration and tool implementation

**Solves:** A2

Restructure `pipeline.py` to pause a second time, after `SchemaArchitectAgent` and `ToolImplementorAgent` complete, before calling `compiler.compile()`. Show the human the generated schemas and function bodies.

```
[Current flow]  decompose → HITL → hydrate → implement → compile

[New flow]      decompose → HITL-1 → hydrate → implement → HITL-2 → compile
```

HITL-2 supports:
- **Approve** — compile as-is.
- **Regenerate node** — re-run `SchemaArchitectAgent` or `ToolImplementorAgent` on a single node.
- **Edit inline** — human edits a function body or schema directly in the UI.

The UI for HITL-2 shows a code diff view for each tool node`s generated function body. This is the checkpoint with the highest signal-to-noise ratio — the LLM's most error-prone output is the one the human sees only here.

### P1.5  Post-compilation verify-and-repair loop

**Solves:** D2

After `compiler.compile()`, before setting `events.status = "done"`, run an import check on the generated project:

```python
result = subprocess.run(
    [sys.executable, "-c", f"import sys; sys.path.insert(0, '{project_dir}'); import run"],
    capture_output=True, text=True
)
if result.returncode != 0:
    # feed traceback back to ToolImplementorAgent for targeted repair
    await _repair(result.stderr, project_dir, tool_implementor)
```

`_repair` parses the traceback to identify the offending file and line, constructs a targeted prompt ("this function body in `atomics/X.py` produces a SyntaxError at line N; here is the body and the error — return a corrected body"), calls `ToolImplementorAgent` for just that node, rewrites the file, and re-runs the import check. Loop up to 3 times. If still failing after 3 iterations, surface the error to the human via HITL rather than silently returning a broken project.

This closes the feedback loop between the compiler's output and the generator's input, covering the single most common failure class (indentation/import errors).

## P2 — Critical Bugs

### P2.1  Fix tool body generation: resolve the indentation ambiguity at the source

**Solves:** B1, C3

Two changes together eliminate the `IndentationError`:

**1. Fix the system prompt** in `tool_implementor.py`'s `_SYSTEM_PROMPT`. Replace:
> "Do not import anything; all stdlib imports must be inside the function body"

with:
> "Every single line of the function body — including import statements — must begin with exactly 4 spaces of indentation. There must be no lines at column 0."

**2. Fix `_normalise_indent`** to be robust against mixed-indentation input by normalising relative to the *first non-empty line* rather than the minimum-common-whitespace baseline:

```python
def _normalise_indent(body: str) -> str:
    lines = body.splitlines()
    # Find the leading indent of the first non-empty line
    first = next((l for l in lines if l.strip()), "    ")
    current_indent = len(first) - len(first.lstrip())
    result = []
    for line in lines:
        if not line.strip():
            result.append("")
            continue
        line_indent = len(line) - len(line.lstrip())
        # Map: current_indent → 4, current_indent+4 → 8, etc.
        relative = line_indent - current_indent
        result.append("    " + " " * max(relative, 0) + line.lstrip())
    # Strip empty leading/trailing lines
    while result and not result[0]:
        result.pop(0)
    while result and not result[-1]:
        result.pop()
    return "\n".join(result) if result else "    raise NotImplementedError()"
```

The prompt fix prevents the problem; the normaliser fix makes it robust against any future LLM drift.

### P2.2  Fix all orchestrator templates in one pass

**Solves:** B2, B3, B4, B5, B6, C6

All six issues are template rendering defects. Fix in one sweep:

**`adk_loop_agent.py.j2`** — emit `max_iterations=10` (configurable constant); document that the sub-agent must call `actions.escalate()` to terminate:
```jinja2
{{ node.name }}_agent = LoopAgent(
    name="{{ node.name }}",
    max_iterations=10,
    sub_agents=[
{%- for imp in child_imports %}
        {{ imp.symbol }},
{%- endfor %}
    ],
)
```

**`adk_coordinator_agent.py.j2`** — the compiler must pass `child_imports` with descriptions; template generates a routing instruction:
```jinja2
{{ node.name }}_agent = LlmAgent(
    name="{{ node.name }}",
    model="gemini-2.0-flash",
    instruction="""{{ node.description }}

Route to the appropriate sub-agent:
{%- for imp in child_imports %}
- {{ imp.symbol }}: {{ imp.description }}
{%- endfor %}
""",
    sub_agents=[...],
)
```

**`adk_llm_agent_stub.py.j2`** — wire `input_schema` into the instruction:
```jinja2
instruction="""{{ node.description }}

Inputs available in session state:
{{ node.input_schema | tojson(indent=2) }}

Write output to session state matching this schema:
{{ node.output_schema | tojson(indent=2) }}
""",
```

**`adk_tool_stub.py.j2`** — add `instruction` to the LLM wrapper and fix enum rendering:
```jinja2
# SANDBOX: This tool executes {{ node.exec_type.value }} logic.
...
{{ node.name }}_agent = LlmAgent(
    name="{{ node.name }}",
    model="gemini-2.0-flash",
    description="{{ node.description }}",
    instruction="{{ node.description }}. Call the {{ node.name }} tool with the appropriate arguments derived from session state.",
    tools=[{{ node.name }}],
)
```

**All composite templates** — fix Jinja whitespace by changing block tags to strip mode:
```jinja2
{%- for imp in child_imports %}
from {{ imp.module }} import {{ imp.symbol }}
{%- endfor %}
```

The compiler must be updated to pass `description` alongside `module` and `symbol` in `_Import` for the coordinator template to use.

## P3 — Quality and Performance

### P3.1  Add sibling and ancestor context to Decomposer and SchemaArchitect calls

**Solves:** C1, C2

Both agents receive nodes without knowing their position in the tree. Add context to each node dict:

For `decomposer.py`:
```python
{"name": n.name, "description": n.description,
 "parent_composition_type": parent.composition_type if parent else None,
 "siblings": [{"name": s.name, "description": s.description} for s in siblings]}
```

For `schema_architect.py`, group nodes by parent and pass the full sibling list for each group, ordered by position in the `SEQUENTIAL` chain:
```python
{"name": n.name, "description": n.description, "exec_type": n.exec_type,
 "preceding_sibling_output_schema": prev.output_schema if prev else None,
 "following_sibling_description": next_node.description if next_node else None}
```

The Schema Architect prompt gains: "For SEQUENTIAL siblings, ensure your output_schema is structurally compatible with the next sibling's expected inputs."

### P3.4  Co-generate `tests/test_imports.py`

**Solves:** D6

Add a final step to `CompilerAgent.compile()` that emits a minimal test file:

```python
# tests/test_imports.py — auto-generated by recur-agent compiler
import importlib, sys, pathlib
sys.path.insert(0, str(pathlib.Path(__file__).parent.parent))

def test_all_atomics_import():
    for path in pathlib.Path("atomics").glob("*.py"):
        if path.stem == "__init__": continue
        importlib.import_module(f"atomics.{path.stem}")

def test_all_orchestrators_import():
    for path in pathlib.Path("orchestrators").glob("*.py"):
        if path.stem == "__init__": continue
        importlib.import_module(f"orchestrators.{path.stem}")
```

This is a compiler-only addition (no new agent). When combined with P1.5 (verify-and-repair), the compiler can run these tests, capture failures, and feed them to `ToolImplementorAgent` for repair — closing the full generate → test → repair loop.

### P3.5  HITL "inputs needed" — surface unknown external dependencies to the human before implementation

**Context:** The `ToolImplementorAgent` (and to a lesser extent the `DecomposerAgent`) must generate code that may depend on external facts the LLM cannot know: the base URL and auth scheme of an internal API, the schema of a proprietary database table, the name of an environment variable used across the codebase, or a domain-specific constant. Without this grounding, generated `EXTERNAL_API` implementations contain placeholder comments (`# CONFIGURE: ...`) that are silently shipped to disk. The human only discovers the gap when running the project.

**Solution:** Add an `ExternalDependencyAgent` that scans the approved atomics — before `ToolImplementorAgent` runs — and identifies which ones have implementation-blocking unknowns, returning a structured list of questions for the human to answer.

**New file: `src/agents/external_dependency.py`**

The agent scans all `EXTERNAL_API` (and flagged `DETERMINISTIC_CODE`) atomic nodes and returns, for each, a list of "inputs needed" — specific questions whose answers are required to generate correct code:

```python
_SYSTEM_PROMPT = """
You are the External Dependency Agent. You receive atomic skill nodes that require
external knowledge to implement correctly.

For each node, identify the specific facts that an LLM cannot reasonably infer and
that a developer would know: API endpoints, authentication methods, database table
names, environment variable names, rate limits, domain-specific constants, etc.

Return a list of precise, answerable questions — not vague requests. Each question
should name the node it belongs to, describe what is needed, and give a short
rationale for why it blocks implementation.

If a node has no external unknowns (e.g., it calls a well-known public API with a
standard contract), return an empty list for that node.
"""
```

Output schema: for each node, a list of `{question: str, context: str}` pairs.

**New field on `SkillNode`:**
```python
external_inputs: dict[str, str] = Field(default_factory=dict)
# key: question text answered by human; value: human's answer
```

**Pipeline integration** — add a new phase between schema hydration and tool implementation:

```
[New flow]  decompose → HITL-1 → hydrate → dependency-scan → HITL-inputs → implement → HITL-2 → compile
```

1. `ExternalDependencyAgent.scan(tool_atomics)` returns `{node_id: [questions]}`.
2. If any questions exist, pause with `events.status = "awaiting_inputs"` and surface them in the UI as a form: one text field per question, grouped by node.
3. Human fills in answers (or skips). Answers are stored in `node.external_inputs`.
4. `ToolImplementorAgent` receives the node dicts enriched with `external_inputs`, and its system prompt gains: "Where `external_inputs` is provided, use those values directly instead of placeholder comments."

**UI change:** The HITL dashboard gains an "Inputs Required" panel that appears only when there are unanswered dependency questions. It looks like a simple form — question label, text input, optional "skip / unknown" checkbox. Answered questions are displayed as read-only in subsequent HITL steps so the human can verify the values were used.

**Why this is P3 and not P1:** The existing `# CONFIGURE: ...` placeholder pattern means the project is still technically generated and runnable (after manual editing). This upgrade makes the generated project complete rather than merely runnable. It is a quality enhancement, not a correctness fix — hence P3, but placed first among P3s because the payoff (correct external wiring) is the most visible quality gap in any non-trivial project.
