# Solution Plan: recur-agent Generated Output Architecture

**Date:** 2026-06-19  
**Status:** P1 + P2 items implemented (2026-06-19). P3 pending.

---

## Overview

The recur-agent platform successfully decomposes requirements into multi-layer skill trees but the compiler discards the tree structure and emits non-functional stubs. This document presents a three-tier solution: fix the workflow and orchestration first (Tier 1), make the output executable (Tier 2), then add type safety and validation (Tier 3).

---

## Priority Order

Implementation priority is independent of the tier grouping. Priority is assigned based on:
- **P1 — Make the flow work:** Without these, the generated project cannot be imported or interacted with at all.
- **P2 — Fix bugs:** Without these, the generated project runs but produces crashes or incomplete results.
- **P3 — Optimize output performance:** These improve quality and type safety but are not blockers.

## Tier 1 — Big Picture: Fix Workflow and Orchestration

These structural changes must land first. They fix Issues 1, 5, and 6—the fundamental shape of the generated output.

### 1.1 Add Composition Type Classification `[P1]` ✅ Implemented

**Problem:** The tree captures *what* to do but not *how* to compose children.

**Solution:** Extend [src/orchestrator/state.py](src/orchestrator/state.py) with a composition classification:

```python
class CompositionType(str, Enum):
    SEQUENTIAL   = "SEQUENTIAL"    # children run one after another in order
    PARALLEL     = "PARALLEL"      # children run concurrently, outputs merged
    LOOP         = "LOOP"          # one child agent runs repeatedly until condition
    LLM_COORDINATOR = "LLM_COORDINATOR"  # LlmAgent decides which child to invoke
```

Add field to `SkillNode`:
```python
composition_type: Optional[CompositionType] = None
```

Update `DecomposerAgent` output schema to require `composition_type` when `node_type == "composite"`. The Decomposer already understands these patterns in its system prompt—it just never records them.

- [src/orchestrator/state.py](src/orchestrator/state.py) — add enum + field
- [src/agents/decomposer.py](src/agents/decomposer.py) — extend output schema + node assignment

### 1.2 Rewrite Compiler as Tree-Recursive Walk (Issue 1 — Critical Fix) `[P1]` ✅ Implemented

**Problem:** Current compiler flattens entire tree into a single root agent with 28 tools/sub-agents.

**Solution:** Recursive compilation algorithm that preserves hierarchy:

```
_compile_node(node) → symbol_name
  if ATOMIC:
    if LLM_PROMPT → render adk_llm_agent.py.j2  → atomics/{name}.py
    else          → render adk_tool.py.j2       → atomics/{name}.py

  if COMPOSITE:
    for child in node.children:
      _compile_node(child)  # depth-first, leaves first
    
    select template by composition_type:
      SEQUENTIAL       → adk_sequential_agent.py.j2
      PARALLEL         → adk_parallel_agent.py.j2
      LOOP             → adk_loop_agent.py.j2
      LLM_COORDINATOR  → adk_coordinator_agent.py.j2
    
    render selected template with direct children
    emit to orchestrators/{name}.py
```

**Generated Project Structure:**

```
{project}/
  atomics/          ← all leaf nodes (tools + LLM agents)
    __init__.py
    ask_clarifying_question.py    (LLM_PROMPT atomic)
    parse_user_response.py        (DETERMINISTIC_CODE atomic)
    fetch_singapore_products.py   (EXTERNAL_API atomic)
    ...

  orchestrators/    ← one file per composite node
    __init__.py
    profile_user_coverage_needs.py
      ├─ imports: ask_clarifying_question_agent, store_indexed_entry
      └─ wires as SequentialAgent
    
    conduct_discovery.py
      ├─ imports: profile_user_coverage_needs, research_singapore_market
      └─ wires as SequentialAgent
    ...

  run.py            ← entry point, imports root orchestrator only
  pyproject.toml
```

Each orchestrator file declares `{name}_agent` and imports only its direct children. The tree structure is now preserved as a hierarchy of ADK wrapping.

**New Templates Required (4 files):**

| File | ADK Class | Description |
|---|---|---|
| `adk_sequential_agent.py.j2` | `SequentialAgent` | Chain children in order |
| `adk_parallel_agent.py.j2` | `ParallelAgent` | Run children concurrently |
| `adk_loop_agent.py.j2` | `LoopAgent` | Loop a child agent + termination condition |
| `adk_coordinator_agent.py.j2` | `LlmAgent` with `sub_agents=[]` | LLM picks which child to invoke |

**Files Changed:**
- [src/agents/compiler.py](src/agents/compiler.py) — complete rewrite of `compile()` method
- [src/compiler_templates/](src/compiler_templates/) — add 4 new templates, update root template

### 1.3 Fix Entry Point (Issue 5 — High) `[P1]` ✅ Implemented

**Problem:** Current `runner.run()` has no user-facing interface.

**Solution:** Update [src/compiler_templates/adk_root_orchestrator.py.j2](src/compiler_templates/adk_root_orchestrator.py.j2) to use ADK's standard CLI:

```python
if __name__ == "__main__":
    # Minimal interactive CLI for testing
    import asyncio
    from google.adk.runners import InMemoryRunner
    
    async def _interactive_loop():
        runner = InMemoryRunner(agent=root_agent)
        print("{{ project_name }} agent running. Type 'exit' to quit.\n")
        
        while True:
            user_input = input("You: ").strip()
            if user_input.lower() in ("exit", "quit"):
                break
            
            # Call the agent with user input
            response = await runner.run_async(
                user_id="user",
                new_message=user_input
            )
            print(f"Agent: {response}\n")
    
    asyncio.run(_interactive_loop())
```

For production, emit a separate `agent.py` module suitable for `adk web` scaffolding.

### 1.4 Fix Schema Hydration Scope (Issue 6 — Medium) `[P2]` ✅ Implemented

**Problem:** Schema hydration only runs on current layer, may miss deeply-nested atomics created in earlier layers.

**Solution:** In [src/orchestrator/pipeline.py:118-127](src/orchestrator/pipeline.py#L118-L127), scan the **entire tree**:

```python
# BEFORE (wrong — only current layer)
approved_atomics = [
    n for n in tree.get_layer_nodes()
    if n.node_type == NodeType.ATOMIC
]

# AFTER (correct — all unhydrated atomics in tree)
approved_atomics = [
    n for n in tree.root.topological_order()
    if n.node_type == NodeType.ATOMIC and n.input_schema is None
]
```

### 2.2 New Agent: PromptEngineerAgent (Issue 4 — High) `[P3]`

**Problem:** LLM agent instructions are just node description + schema. No prompt engineering; inputs not wired.

**Solution:** Add a new agent that generates production-grade instructions.

**New File:** [src/agents/prompt_engineer.py](src/agents/prompt_engineer.py)

```python
class PromptEngineerAgent(BaseAgent):
    """Generate well-engineered instructions for LLM atomic nodes."""
    
    async def engineer(self, nodes: list[SkillNode]) -> list[SkillNode]:
        """Enhance instruction field for each LLM node."""
        node_dicts = [
            {
                "name": n.name,
                "description": n.description,
                "input_schema": n.input_schema,
                "output_schema": n.output_schema,
            }
            for n in nodes
        ]
        
        message = await self._call(
            messages=[{"role": "user", "content": (
                "Write production-grade instructions for each LLM node. "
                "For each node, include:\n"
                "1. Description of the task\n"
                "2. Explicit input parameter documentation\n"
                "3. Step-by-step reasoning the model should follow\n"
                "4. Output format with field descriptions and examples\n"
                "5. Any constraints or guardrails\n"
                "\nNodes:\n" + json.dumps(node_dicts, indent=2)
            )}],
            output_schema={...}  # array of {"instruction": "..."}
        )
        
        # Unwrap and assign results
        for node, result in zip(nodes, results):
            node.instruction = result["instruction"]
        
        return nodes
```

### 3.1 Composite Node Schema Hydration (Issue 7) `[P3]`

**Problem:** Composite nodes have null input/output schemas; data contracts between layers undefined.

**Solution:** Extend `SchemaArchitectAgent` with a bottom-up pass for composites.

Add method to [src/agents/schema_architect.py](src/agents/schema_architect.py):

```python
async def hydrate_composites(self, tree: SkillTree) -> None:
    """Synthesize I/O schemas for composite nodes (bottom-up)."""
    # Collect all composites, sorted by depth descending (leaves first)
    all_composites = [
        n for n in tree.root.topological_order()
        if n.node_type == NodeType.COMPOSITE
    ]
    
    for composite in all_composites:
        # Gather child output schemas
        child_outputs = [
            {"name": c.name, "schema": c.output_schema}
            for c in composite.children
        ]
        
        # Ask agent to synthesize composite's I/O
        result = await self._call(
            messages=[...],  # composite description, composition_type, child outputs
            output_schema={...}  # {"input_schema": ..., "output_schema": ...}
        )
        
        composite.input_schema = result["input_schema"]
        composite.output_schema = result["output_schema"]
```

**Pipeline Integration:** New phase in [src/orchestrator/pipeline.py](src/orchestrator/pipeline.py) after all layers decomposed:
```python

    logger.info("Hydrating composite node schemas...")

```

### 3.2 Data Flow Validation Pass `[P3]`

**Problem:** No type checking between composed nodes; mismatches masked.

**Solution:** Light validation pass before compilation (pure Python, no LLM).

**New File:** [src/agents/validator.py](src/agents/validator.py)

```python
class DataFlowValidator:
    """Check structural type compatibility between connected nodes."""
    
    def validate_tree(self, tree: SkillTree) -> list[str]:
        """Return list of warnings (non-fatal, for logging)."""
        warnings = []
        
        for node in tree.root.topological_order():
            if node.node_type != NodeType.COMPOSITE:
                continue
            
            if node.composition_type == CompositionType.SEQUENTIAL:
                # Check output[n] → input[n+1] compatibility
                for i in range(len(node.children) - 1):
                    out_schema = node.children[i].output_schema
                    in_schema = node.children[i+1].input_schema
                    if not _schemas_compatible(out_schema, in_schema):
                        warnings.append(
                            f"{node.name}: child[{i}] output incompatible with "
                            f"child[{i+1}] input"
                        )
        
        return warnings
```

**Pipeline Integration:** In [src/orchestrator/pipeline.py](src/orchestrator/pipeline.py) before compilation:
```python
validator = DataFlowValidator()
warnings = validator.validate_tree(tree)
for w in warnings:
    logger.warning(w)
```

## Notes

- **Backwards Compatibility:** These changes are not backwards-compatible with existing blueprints (new `composition_type` field on `SkillNode`). Existing blueprints must be re-run through the updated Decomposer before compilation; the compiler raises `ValueError` on composite nodes with `composition_type=null` rather than defaulting to SEQUENTIAL.
- **Tool stub wrapping:** `adk_tool_stub.py.j2` now emits both a `FunctionTool` and a wrapping `LlmAgent` so tool atomics can be used as `sub_agents` in orchestration templates (ADK `SequentialAgent/ParallelAgent/LoopAgent` all require `BaseAgent` sub_agents, not raw callables).
- **Unified symbol convention:** All compiled nodes (atomic tools, LLM agents, composites) export a `{name}_agent` symbol. Orchestration templates always import and reference `{name}_agent`.
- **Graceful Degradation (P2):** If PromptEngineerAgent or ToolImplementorAgent fail, the pipeline falls back to stub templates. Generated code will be less capable but still importable.
- **Future Extensions:** The `composition_type` field enables future work on automated orchestration optimization (e.g., convert SEQUENTIAL→PARALLEL where safe).
